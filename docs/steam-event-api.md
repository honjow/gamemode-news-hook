# Steam Event API Research

Research notes on Steam's event/announcement API internals, gathered from
API responses and CEF remote debugging on SteamOS and ChimeraOS devices.

## OS update branches

Steam exposes OS update branches via `SteamClient.Updates.GetOSBranchList()`.

On stock SteamOS, 7 branches are defined:

| eBranch | sRawName | Description |
|---------|----------|-------------|
| 1 | `rel` | Stable |
| 2 | `rc` | Release Candidate |
| 3 | `beta` | Beta |
| 4 | `bc` | Beta Candidate |
| 5 | `preview` | Preview |
| 6 | `pc` | Preview Candidate |
| 7 | `main` | Development |

Other Linux gaming distributions may define their own branch options
(desktop mode selection, build variants, etc.) while keeping `rel`,
`beta`, and `main` as system update channels.

`SteamClient.Updates.GetCurrentOSBranch()` returns the active branch,
e.g. `{"eBranch": 1, "sRawName": "rel"}`.

## Event API channel tags

The event API (`ajaxgetadjacentpartnerevents`) uses `require_tags` to
filter announcements by update channel. For appid 1675200 (Steam Deck),
only three tags have actual content:

| API tag | Branch |
|---------|--------|
| `stablechannel` | rel (Stable) |
| `betachannel` | beta (Beta) |
| `previewchannel` | preview (Preview) |

The remaining branches (`rc`, `bc`, `pc`, `main`) have no dedicated
event tags and likely share announcements with their parent channel.

Requests without `require_tags` return a mixed feed dominated by the
latest beta updates.

## How Valve sets channel tags

Channel tags (`stablechannel`, `betachannel`, etc.) are stored in the
`announcement_body.tags` array. Example from Valve's SteamOS announcements:

```
stable:  tags: ["patchnotes", "stablechannel", "mod_reviewed", ...]
beta:    tags: ["patchnotes", "betachannel", "previewchannel", "mod_reviewed", ...]
no tag:  tags: ["patchnotes", "mod_reviewed", ...]
```

These tags are **auto-generated by Steamworks** when a developer links an
event to a specific build branch via the "Link to Build" button in the
Steamworks event editor (documented in
[Event Type: Small Update / Patch Notes](https://partner.steamgames.com/doc/marketing/event_tools/type_patchnotes)).

This feature is only available for **app-level announcements** published
through Steamworks. Community group announcements do not have access to
the build linking system, so their `announcement_body.tags` is always
an empty array.

## Event data structure

Key fields from `ajaxgetadjacentpartnerevents` response (per event):

| Field | Description |
|-------|-------------|
| `event_name` | Display title (used on outer card) |
| `event_type` | Category number (12 = small update/patch notes, 28 = community announcement) |
| `event_notes` | Free text (Valve uses "see announcement body") |
| `build_branch` | Linked build branch (empty for community groups) |
| `build_id` | Linked build ID (0 for community groups) |
| `jsondata` | JSON blob with localized images etc. (null for community groups) |
| `announcement_body.headline` | Subtitle (shown in expanded view) |
| `announcement_body.body` | BBCode content |
| `announcement_body.tags` | Array of tags (empty for community groups) |
| `announcement_body.posttime` | Unix timestamp |

## Per-channel announcements for community groups

Since community group announcements lack the Steamworks build-linking
mechanism, per-channel filtering requires a custom marking scheme.

### Implemented: title prefix

The chosen approach uses a single-character title prefix (e.g. `[S]`,
`[B]`, `[P]`) mapped to OS branch names via `[channel_prefix]` config:

```ini
[channel_prefix]
S = rel
B = beta
P = preview
```

Behavior:
- `[S] System Update` → shown only on `rel` branch, displayed as `System Update`
- `[B] Beta Update` → shown only on `beta` branch
- `System Update` (no prefix) → shown on all branches
- Empty `[channel_prefix]` section → feature disabled

The prefix is stripped at display time by the JS XHR hook. On the Steam
community page the prefix remains visible (e.g. `[S] System Update`).

OS branch is detected at injection time via
`SteamClient.Updates.GetCurrentOSBranch()`. Filtering happens in JS on
each intercepted request, so newly published announcements take effect
immediately without re-injection.

### Alternatives considered

- **Hidden BBCode marker in body** — tested; all BBCode tags render
  visibly on the community page, no way to hide content.
- **Zero-width character encoding** — inconvenient to write when
  composing announcements.
- **Separate community groups** — clean but high maintenance overhead.

None of these are needed if the same announcements should be shown
across all update channels.
